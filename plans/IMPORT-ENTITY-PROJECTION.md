> 制定日期：2026-07-15
> 定位：把整合包声明内容（import）的部署投影从软链接改为实体文件以让游戏直接读取目录型资源，同时把工作副本层从独立的 live 目录重定义为「import 在 build 上的虚拟投影」，并相应改造快照、workspace 与整合包更新流程
> 当前状态：未实施

## 计划

### 背景与动机

整合包声明内容（import）目前先实体复制进一个独立的 live 工作副本目录，再从 live 软链接进 build。这套软链接投影几乎全是优点——低成本、自带来源指向、重布局快——只有一处致命缺点：游戏不识别某些路径上的软链接。目录型资源包的音乐、贴图、模型、语言等资源因此加载不出来，就是这个架构缺点的症状。

包（mod、库）和持久化内容（存档等）走软链接没有这个问题——游戏对它们的读取/写入本就走链接，所以这两类投影不变。需要改的只有 import：从软链接改为实体文件。

但一旦 import 变成实体文件，原先软链接「免费」提供的工作副本语义就失去了载体——独立的 live 目录既是 import 的暂存副本，又是游戏透过链接写回的工作状态，还承载了快照、workspace 变更追踪与整合包更新的「整体清空再重建」语义。简单删掉 live 会让这些能力失去落点，也躲不开「有则不管」的部署规则挡住整合包内容更新传播的坑。

因此核心不是「换一种投影」，而是给工作副本层换一个更合适的载体：live 不再是一个目录，而是「import 在 build 上的虚拟投影」——它的文件清单来自 import，它的文件状态取自 build。

### 目标

- import 投影到 build 为实体文件，使游戏能原生读取目录型资源包等此前走软链接读不到的内容。
- 包与持久化投影维持软链接不变，游戏对它们的读写行为不变。
- 工作副本（live）作为概念保留，但重定义为虚拟投影：清单取自 import，状态取自 build，不再有独立目录。
- 整合包更新采用替换式语义：清除旧工作副本后从新 import 重新投影，由更新前的快照充当后悔机制。
- 快照继续覆盖工作副本状态（改从 build 的 import 投影子集读取），使替换式更新可逆。
- workspace 的变更追踪与 stage/restore 作用在重定义后的工作副本上。

### 非目标 / 不做的事

- **不持久化任何部署清单/ledger。** 需要证明 import 放过什么时，由 import 目录本身当场推导，不另存记录。本计划原标题中的「ledger」即此义，已否决。
- **不改包与持久化的投影方式。** 它们仍走软链接。
- **不在整合包更新中保留工作副本改动。** 替换式更新刻意丢弃它们，由快照兜底。
- **不兼容旧的软链接式 import 投影。** 检测到遗留部署直接报错并引导重置；系统可重建性是首要要求，重置即可清理。
- **不在本计划中重做部署管线的内部机制。** 部署管线的分阶段重构由关联的独立计划落实；本计划只固定投影模型与受其影响的各层边界。
- **不优雅处理每一种歧义冲突。** 确定性情形直接处理，歧义情形报错并交给重置。

### 关键决策与取舍

- **import 走实体文件，包与持久化走软链接。** 按游戏是否跟随链接读取该类内容来切分。软链接除了游戏识别这一点外都是优点，只修坏的那部分。
- **live 重定义为虚拟投影，而非保留或删除目录。** 清单 = import，状态 = build。这样既保住了工作副本概念以及建在其上的全部能力（快照、workspace、stage），又不需要独立暂存目录，也不需要持久化清单。这是整份计划的核心动作。
- **整合包更新用替换式 + 快照兜底，不做差分删除。** 整体清空旧工作副本再从新 import 重建，避免逐项判冲突，也绕开「有则不管」挡住内容更新传播的问题。代价是工作副本改动在更新中丢失，由更新前快照还原。
- **不引入持久化 ledger。** import 的来源证明在需要时（部署存在性检查、更新清除步、快照记录、workspace 查询）由 import 目录现算。另存清单会退化成其他启动器的 manifest 追踪模式，多一份易失配的记录。
- **import 实体投影走独立的收集/应用通道，不复用库与资源的下载/解压通道。** 往 build 放实体文件的语义（按存在性补、参与与持久化软链接的优先级仲裁）与下载、解压不同，需要一条与现有软链接对账器平行的专属机制。
- **检测到遗留的软链接式 import 投影直接报错。** build 中本应放 import 实体的路径若现存软链接，判为遗留部署，报错并提示重置实例；不为旧机制维护迁移路径。
- **同一路径被多方声明时优先级不变。** 持久化 > import > 包的仲裁顺序保留，仅 import 的交付方式从软链接变为实体文件。

### 影响面

- **部署投影模型** —— import 投影交付方式由软链接改为实体文件；包与持久化投影不变；三方在 build 内的优先级仲裁保留。
- **工作副本层（live）** —— 从独立目录降格为虚拟投影：清单取自 import，状态取自 build。所有依赖 live 的能力随之改造。
- **整合包更新流程** —— 删除旧 live 目录的动作变为「清除 build 中旧的 import 工作副本」；更新语义为替换式（先清后建）。
- **快照** —— 记录范围调整：仍记录 import、持久化与 setup，外加 build 中 import 投影子集（即虚拟 live 的工作副本状态）；还原时以 import 清单界定「需删除的多余项」范围，避免误伤 build 中的非托管内容。
- **workspace 变更追踪** —— 检测入口从遍历 live 改为遍历 import、按图索骥到 build 查存在性与差异；stage（工作副本→import）与 restore（import→工作副本）语义不变，操作对象从 live 目录改为 build 投影。
- **整合包导出** —— 仍导出干净的 import；用户可经 workspace 把工作副本 stage 进 import 后再导出。
- **实例重置** —— 重置的清理动作去掉 live 目录。
- **遗留部署兼容** —— 旧软链接式 import 投影的检测改为报错并引导重置。
- **部署管线内部机制** —— 由关联的独立计划落实，本计划只定模型与边界。

### 验收标准

#### 正常部署

- 新实例部署后，import 声明的内容（含目录型资源包的音乐、贴图、模型、语言）在 build 中为游戏可直接读取的实体文件。
- 包与持久化内容仍以软链接投影，游戏对它们的读写行为不变。
- 同一路径被持久化与 import 同时声明时，持久化优先，import 不在该路径放置实体文件。
- 日常重新部署只补缺失的 import 实体文件，已存在（含被游戏改过）的不动。

#### 整合包更新

- 更新后，build 中对应旧 import 的工作副本被清除，新 import 的内容被重新投影为实体文件。
- 更新不触碰持久化投影（存档等）与包投影，也不触碰 build 中非 import 路径的内容。
- 用户对工作副本的改动在更新中会丢失，且可由更新前的快照还原。

#### 漂移与 workspace

- workspace 能以 import 为清单、build 为状态来源，呈现工作副本相对 import 的变更，并能 stage（提升进 import）或 restore（丢弃回 import）。

#### 快照与恢复

- 快照能记录并还原 import、持久化、setup，以及 build 中 import 投影的工作副本状态。
- 还原快照不误删 build 中非 import 路径的内容。
- 检测到遗留的软链接式 import 投影时，部署报错并提示重置实例；重置后可从来源干净重建。

### 备选方案备案

- **持久化 ledger 记录已放置实体文件清单。** 否决。会退化成其他启动器的 manifest 追踪模式；且来源证明在需要时可由 import 目录现算，不必持久化存储，多一份记录反而引入失配与维护负担。本计划原标题中的「ledger」即此义，已否决——Prism 等「按新 manifest 删旧文件」的路径正是这类追踪的典型翻车源。
- **import 与持久化都改实体投影。** 否决。持久化（存档等）依赖软链接的写回间接——游戏透过链接直接写进持久化目录；改实体后游戏写的是 build 副本，下次部署会被持久化源覆盖而丢档。只有游戏需直接读取的 import 需要实体化。
- **保留 live 目录，仅把其投影方式改为实体。** 否决。那样会同时存在 live 目录与 build 实体副本两份工作副本状态，且 live 目录的清理/快照逻辑仍要维护，复杂度不降反增。虚拟投影（清单 = import，状态 = build）更统一。
- **整合包更新用 old−new 差分删除而非整体替换。** 否决。差分删除需逐项判冲突，且解不开「import 内容变了但 build 有则不管」导致更新传不进去的传播问题；整体清空旧工作副本再重建更简单，丢失风险由快照兜底。
- **兼容旧软链接投影（自动迁移）。** 否决。系统可重建性是首要要求，重置即可清理遗留；为旧机制维护迁移路径违背「完美系统」取向，且迁移判定本身易错（损坏、改向、来源不可信的旧链接难以安全认定）。
- **快照记录整个 build 以覆盖工作副本。** 否决。build 混有日志、截图、崩溃报告、包链接、持久化链接等大量非托管内容，全量记录既臃肿又把快照变成 build 镜像；以 import 清单界定工作副本范围，只记 build 中 import 投影子集，既覆盖工作副本又不引入噪音。
- **硬链接投影 import。** 否决。硬链接无法可靠表达来源、来源删除后无法证明所有权、build 原地修改可能反向污染来源。

### 关联计划

部署管线的分阶段重构（投影收集/应用通道、与软链接对账器平行的实体投影机制、遗留检测报错点）由 `DEPLOYMENT-PROJECTION-REWORK.md` 落实。本计划只固定投影模型与各受影响层的边界，不规定管线内部实现。
