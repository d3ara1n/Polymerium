# plans/

这里存放**任务计划文件**，彼此独立。有些共享某个宏观目标，那是弱关联，正文提一句即可。本目录不维护索引、依赖图、任务清单或分类体系——刻意保持随意，随时新建、随时归档，目录自身的维护成本应趋近于零。

每个文件分两节，先后落盘：**计划**（做什么、为什么、做到什么程度，即 WHAT）和**方案**（怎么实现，即 HOW）。

## 总过程

一份文件从想到到归档，走这条线：

1. **想到要做什么** → 和 agent 讨论，产出**计划**，写入文件。此时文件只有计划一节。
2. （可能隔很久）**正式打算实施** → 读计划，和 agent 讨论，对照**当下的真实代码**产出可落地的**方案**，作为 `## 方案` 段追加到文件末尾。
3. **实施** → 遇障碍、解障碍，方案随之修订。
4. **完成** → 按实际实施过程把方案定稿写回（可含代码、符号、文件等实施细节——因为是改完代码后才落盘，记录的是既成事实与选择路径，不是预测）。
5. 不再需要留作参考 → `mv` 进 `archived/`。

## 计划（WHAT）

计划是文件的主体，也是个**引子**——它让接手者（包括未来的自己和实施时的 agent）读懂要做什么、为什么、边界在哪、怎么算完成。

**计划里禁止的东西：**

- **禁止任何代码** — 片段、类型签名、特性标注、正则、SQL、配置键、JSON 片段，一律不写。决策用自然语言讲清"选了什么、为什么"。
- **禁止文件:行号表、禁止穷举符号清单** — 这类引用随重构腐烂，且诱导照搬。影响面用**能力/领域**描述（"包标识的解析与构建层"、"规则选择器枚举"），精确落点交给实施时 grep 当下代码。
- **禁止规定实现机制** — "用 `System.Uri` 解析"、"加一个兼容 shim"、"起一个并行新项目"是方案级决策，不属于计划。计划只说"输入要能正确解析成包描述"、"旧格式数据要能无缝读入并在下次写入时迁移"。
- **禁止进度标记** — ⏳🔨✅ 是方案才有的（见下）。计划只描述目标，不追踪施工。

违反以上任一条，应视为**过时草稿**而非可交接的计划。

**成品视角，不写过程。** 计划是给没参与过决策的接手者读的成品，不是讨论纪要。只写最终状态——决策是什么、影响面在哪、怎么验收；不写"原来是 A 改成 B""早期草案如何"这类决策变化历史。

一份计划通常包含：

- **背景与动机** — 现状、为什么得改
- **目标 / 非目标 / 不做的事** — 边界三处各说一遍，挡住 scope creep
- **关键决策与取舍** — 方向性选择及理由，不是代码翻译
- **影响面** — 按**能力/领域**描述波及范围，让接手者知道爆炸半径
- **验收标准** — **行为级**场景表：用户/系统能观察到什么，而非内部怎么实现（"旧配置能读入并在下次保存时迁移到新格式" ✓；"Purl setter 调用 MigrateFromPurl" ✗）
- **备选方案备案** — 见下

### 备选方案备案

计划的核心价值之一。每个关键决策点，把**未采纳但可行**的备选连同**否决理由**一起留底。计划的生命周期里，同一个被否决的方案会被反复重新提出；留底一次，就断了重复讨论的回路。它记录的是"被否决的方案 + 否决理由"这个静态决策，不是讨论过程。

## 方案（HOW）

方案在**实施时**产出，作为 `## 方案` 段追加到文件末尾。它对照当时的真实代码，记录实际怎么做的——机制、代码结构、符号、文件、序列化与 API 选择都写在这里。

**方案里可以写代码和实施细节。** 这正是它和计划的区别：计划不写 HOW，是因为写的时候还不能确定；方案写 HOW，是因为已经做出来了（或至少敲定了改动）。方案在实施期间是活的，随障碍解决而修订；完工时定稿，此后不再改（除非返工）。

**分阶段与进度。** 不能一步完成的工作，在方案里按阶段拆分，用标记追踪——这是方案独有的，计划里不出现：

- ⏳ 待实施
- 🔨 进行中
- ✅ 已完成

完成一个阶段后，把「实际怎么做的」写回该阶段描述；若实施时偏离了计划原意、补了细节、或敲定了原本悬而未决的选择，一并记录。阶段状态是接手者判断「做到哪了」的唯一依据——**进度必须落在方案里，不要让接手者去读代码反推**。

## 头部元信息

每份文件开头用引用块声明：

```
> 制定日期：YYYY-MM-DD
> 定位：<一句话说清这是什么任务>
> 当前状态：未实施 | 实施中 | ✅ 已实施
```

`当前状态` 反映方案的生命周期：

- **未实施** — 只有计划一节，还没动工
- **实施中** — 方案已开写、正在施工
- **✅ 已实施** — 方案定稿、施工完成

实施进度**不可信**：文件会滞后于代码——实施了却忘记回填状态、或忘记归档，都是常态，所以判断有没有落地要以代码为准，别凭状态字样下结论。确认已实施时有两条收尾：还要被别的文件引用、想留作参考，就标 `✅ 已实施`；不再需要了，`mv` 进 `archived/` 即表示 done。方案里的阶段标记（⏳🔨✅）是实施当下同步回写的，比头部状态可信。

## archived/

`archived/` 是坟墓。归档进去的东西可能是已完工、半途废弃、或当初随手写下的草稿——**一律没有参考价值，不要读**。任何指向归档文件的链接视为已失效，读到时跳过，不要顺藤摸瓜进去。

## 引用规范

- **能力/领域描述优先，不用文件:行号** — 写"Profile 中记录包来源的字段"，不写 `Profile.cs:43`。前者不腐烂，后者会。方案部分例外：方案记录既成事实，可用具体符号与路径。
- **不要引用归档文件** — 归档即不可信。若一份计划的依据来自已归档的内容，把依据本身搬过来或改写自洽，不要留指向 `archived/` 的链接。

## 日常操作

- **新建** — `plans/<NAME>.md`，NAME 用大写连字符描述性名字（如 `LOCKDATA-MODERNIZATION`）
- **归档** — `mv plans/X.md plans/archived/`。归档即进坟墓，无需更新指向它的引用
- **断链** — 发现指向归档的链接，删除或改写自洽，**不要**把链接修到 `archived/` 下
