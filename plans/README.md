# plans/

这里存放**独立任务**的蓝本。每份计划的目标是让接手者读完就能直接施工——不必重走当初定方案的讨论，不必从代码反推进度。

计划之间彼此独立。有些计划共享某个宏观目标，那是弱关联，在正文里提一句即可。本目录不维护索引、依赖图、任务清单或任何分类体系——刻意保持随意，随时新建、随时归档，目录自身的维护成本应趋近于零。

## archived/

`archived/` 是计划坟墓。归档进去的东西可能是已完工、半途废弃、或当初随手写下的草稿——**一律没有参考价值，不要读**。任何指向归档计划的链接视为已失效，读到时跳过，不要顺藤摸瓜进去。

## 结构约定

### 头部元信息

每份计划开头用引用块声明：

```
> 制定日期：YYYY-MM-DD
> 定位：<一句话说清这是什么任务>
> 当前状态：草案 | 蓝本
```

`当前状态` 两态，描述计划的成熟度：

- **草案** — 方向仍在探索，不可照此施工
- **蓝本** — 方案已定，可直接编码

实施进度不在这里追踪——读代码就知道动没动工。分步骤计划的进度走下一节的阶段标记。

### 正文

一份可施工的蓝本通常包含：

- **背景与动机** — 现状、为什么得改。问题摆清，方案才有依据
- **目标 / 非目标 / 不做的事** — 边界三处各说一遍，挡住 scope creep
- **核心设计** — 关键决策与取舍，不是代码翻译
- **改动面** — 用 **文件:行号** 列出要动的地方，接手者不必重新 grep
- **验收标准** — 场景表，定义完成态

### 备选方案备案

这是计划的核心价值之一。每个关键决策点，把**未采纳但可行**的备选连同**否决理由**一起留底。计划的生命周期里，同一个被否决的方案会被反复重新提出；留底一次，就断了重复讨论的回路。

## 分步骤计划

不能一步完成的任务按阶段拆分，每个阶段独立追踪进度。阶段状态是接手者判断「做到哪了」的唯一依据——**进度必须落在计划里，不要让接手者去读代码反推**。

阶段用标记表达状态：

- ⏳ 待实施
- 🔨 进行中
- ✅ 已完成

完成一个阶段后，回写两件事：

1. 把该阶段标记改为 ✅
2. 把「最终采用的方案 / 实际怎么做的」写回该阶段描述——若实施时偏离了原设计、补了细节、或敲定了原本悬而未决的选择，一并记录

回写之后，该阶段对下一个接手者就是自洽的现状说明，无需再读代码确认。

## 引用规范

- **文件:行号优于文字描述** — `Profile.cs:43` 比「Profile 里的 Source 字段」高效十倍，且实施前可一键核对
- **不要引用归档计划** — 归档即不可信。若一份计划的依据来自已归档的内容，把依据本身搬过来或改写自洽，不要留指向 `archived/` 的链接

## 日常操作

- **新建** — `plans/<NAME>.md`，NAME 用大写连字符描述性名字（如 `LOCKDATA-MODERNIZATION`）
- **归档** — `mv plans/X.md plans/archived/`。归档即进坟墓，无需更新指向它的引用——按引用规范，本来就不该存在这种链接
- **断链** — 发现指向归档的链接，删除或改写自洽，**不要**把链接修到 `archived/` 下
