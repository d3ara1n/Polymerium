# 导入入口统一：NewInstancePage 承接在线导入

> 制定日期：2026-07-19
> 定位：体验任务，把 NewInstancePage 从「只认本地文件」升级为「本地 + 在线统一配置后创建」。
> 当前状态：构想（未实施）。下面描述的是理想形态与想要的效果，不是已落地的方案。
> Jira：[POLY-135](https://d3ara1n.atlassian.net/browse/POLY-135) —「导入需要有不同的对话框，提供拖入资源与直接输入 pref/url/文件路径」
> 依赖：已完成的导入对话框与 URL 识别前置（见 §3）。

---

## 1. 背景与动机

POLY-135 要的是一个**统一的导入入口**：一个对话框里既能拖文件，也能直接粘贴 `pref://`、http(s) 链接或本地路径。前置工作（对话框本体、本地三态分类、core 的 URL→pref 识别）已经做完并提交，**但对话框还没有接到 NewInstancePage**——目前 `OpenImportDialog` 仍是老的文件选择器路径，新对话框游离在流程之外。

真正缺的一环，是把对话框产出的三种结果（本地文件 / pref / 网址）统一接进 NewInstancePage 的「配置后创建」流程，让在线导入也能像本地导入一样：先看到包的图标、名字、版本，填上自定义名字和图标，再点创建。

现在的 NewInstancePage 只为「本地文件导入」和「空实例」两种来源设计；在线导入（pref / 网址）没有落脚点。本任务补上这个落脚点。

---

## 2. 想要的效果

### 2.1 三分支统一进入配置页

对话框确认后，无论结果是本地文件、pref 还是网址，都回到 NewInstancePage 进入**配置态**，让用户在创建前预览并改写名字、图标。三条路径在配置页里的唯一区别是「信息从哪来、能改到什么程度」。

### 2.2 ModpackSource 抽象：LocalPack 与 OnlinePack

用一个统一的来源抽象（暂称 `ModpackSource`）承载进入配置页的包信息，按来源派生两个形态：

- **LocalPack** —— 本地文件导入得到的包。行为对标现在的 `ImportedPack`：包内容已经在本地解出，图标、名字、版本、加载器、包数量都拿得到，且**版本文本是真实可信的**（直接来自包内 profile）。
- **OnlinePack** —— 在线 pref / 网址对应的待导入包。创建前**还没有真正下载**，配置页展示的图标、名字来自仓库元数据（通过 pref 解析或网址识别得到），但**拿不到确切的 Minecraft 版本**——版本要等下载、解包后才确定。

配置页根据当前来源是哪一种，决定哪些字段可编辑、哪些字段怎么显示（见 §2.3、§2.4）。

### 2.3 DisplayName 与图标：两种来源都要可改

无论本地还是在线，用户都要能在配置页填自定义名字（DisplayName）和拖入自定义图标。这两个值是**用户对即将创建的实例的命名与外观意愿**，应覆盖包自带的默认值。

### 2.4 版本框的禁用与显示策略

- **LocalPack**：版本框可显示真实版本（现状如此），按现状处理。
- **OnlinePack**：版本框必须禁用——因为版本在下载前未知。但**禁用状态下若仍显示某个文本（例如默认版本号）会误导用户**，让人以为那就是包的版本。需要想清楚禁用时到底显示什么（留空？占位提示？占位要传达「版本将在下载后确定」），避免禁用框里的文本造成误解。

### 2.5 创建时的名称与图标覆盖

创建实例时，用户填的 DisplayName 与图标要真正落到实例上、覆盖包自带的默认值。这要求创建/安装链路支持**名称与图标的覆盖入口**。现状的 `InstanceManager.Install` 没有这层覆盖能力，这是本任务里**最大的一块改造**（见 §4）。

### 2.6 NewInstancePage 的进度展示

在线导入（以及顺带的本地导入）在创建阶段涉及下载/解包，是有耗时的操作。NewInstancePage 目前没有任何进度展示能力——点创建后用户面对的是一个无反馈的等待。本任务要为 NewInstancePage 增加进度展示（下载进度、阶段提示等）。本地导入虽然快，但也一并接入进度展示，作为 UX 的整体提升，而非只服务在线分支。

## 3. 已完成的前置（已提交，可直接复用）

以下资产已经落地，本任务直接消费，不重做：

- **`ModpackImporterDialog`** —— 完整的导入对话框：拖放区 + 文本输入框 + 显式 Validate（对勾）按钮 + 结果/提示文本。拖入文件或粘贴路径后点 Validate 完成本地无网络分类。
- **`ModpackImporterResult`** —— 三态 union：`File(string path)` / `Pref(PackageIdentifier)` / `Uri(Uri value)`。
- **`ModpackUrlDetectionHelper`** —— 纯本地、无网络的输入分类器，把输入归到三态之一。
- **`RepositoryAgent.RecognizeAsync(Uri)`**（core） —— 把 http(s) 网址解析成 `PackageIdentifier`，用于把 `Uri` 分支归约成 `Pref` 分支。

## 4. 已知难点与改造规模

- **`InstanceManager.Install` 的名称/图标覆盖（大改造）** —— 这是本任务最重的部分。当前 `Install(key, label, ns, pid, vid)` 下载、解包后直接用包自带的 profile 注册实例，没有入口让调用方覆盖名字和图标。要支持覆盖，需要触碰 core 的 install→register 链路。
- **NewInstancePage 的状态机扩展** —— 从「无来源 / 本地文件」两态，扩展为「无来源 / LocalPack / OnlinePack」，并用统一的来源抽象驱动 UI（图标、名字、版本框的可编辑性与显示）。遵循项目规则：**不要把一个逻辑状态拆成多个 IsVisible 各自门控**，用来源类型单一判定 + 视图按类型切换。
- **CreateAsync 的异步化** —— 在线导入的创建要走下载流程（带 tracker），CreateAsync 当前是同步注册，需要演进为能承载在线下载/进度的异步创建。

## 5. 调研得知的注意事项（给接手者）

- **`Profile.OVERRIDE_MODPACK_NAME` 是导出语义，不是创建时的显示名覆盖入口**。它服务于导出场景，实例创建后才生效、给导出用。想用它做「创建时覆盖名字」是语义错配，之前尝试过、判定走不通，别再走这条。
- **本地导入的 key 与 Name 来源不一致**：没填 DisplayName 时，key 回退用 VersionName 生成 slug，而 `profile.Name` 用包自带的名字——两者来源不同。这点在统一名称覆盖设计时要一并厘清（见 §6 的开放问题）。
- **`InstanceManager.Install` 是带 tracker 的异步流程**：它有 `InstallTracker`、`InstanceInstalling` 事件、进度流。在线导入天然适合走它；难点只在「如何把用户的名字/图标覆盖注入这条链路」，而不是「如何发起在线下载」本身。
