# Java Runtime 自愈：区分用户 JRE 与 bundle JRE 的文件清单校验

> 制定日期：2026-07-13
> 定位：修复部署 pipeline 里 Java runtime 唯一无法自愈的断层——让 bundle JRE（pipeline 自己装的）像 library/asset 一样走无条件文件清单校验、损坏文件能自动重下；用户自配的 JRE 保持不管。
> 当前状态：未实施

---

## 背景与动机

### 现状

整套部署 pipeline 的设计目的，是让实例依赖的所有文件在启动前能自愈——每个 library、asset、package 都在清单生成与固化阶段过一遍 hash 校验，不匹配就重新下载。这套机制让缓存损坏、下载中断、CDN 篡改都能在下一次启动时静默修复。

唯独 Java runtime 不在这套自愈闭环里。引擎定位 Java 的调用点只做"入口文件是否存在"的判定：只要 bundle JRE 的可执行入口还在，就认为 Java 就位、跳过 runtime stage 的整个重装分支。它既不重新校验 bundle JRE 的文件清单，也不实际验证 Java 能否启动。

### 问题

bundle JRE 的某个文件损坏（典型：核心共享库被截断），入口文件仍存在，于是定位成功、runtime stage 跳过，损坏文件永远不会被重新校验或重下。实例启动时 JVM 在加载阶段直接崩溃（动态库加载失败），且崩溃发生在游戏日志系统初始化之前——既无游戏日志、也无任何文件产出，极难定位。实际触发过：远端下载的 bundle JRE 的核心库截断，导致所有使用该 Java 大版本的实例无一例外启动即崩，而 pipeline 全程无报错。

这是整套 pipeline 里唯一用"文件存在"冒充"文件可用"的地方——其余文件的可用性都由 hash 校验保证。

### 根因

bundle JRE 归 pipeline 管（它是 pipeline 自己装的），却没有被纳入 pipeline 的文件清单自愈；而 pipeline 又无法把它和用户自配的 JRE 区分开，于是只能用最弱的"入口文件存在"判定一刀切。区分开两者，才能对 bundle JRE 施加和 library/asset 同等强度的校验，对用户 JRE 保持不干预。

## 目标 / 非目标 / 不做的事

**目标**

1. 部署 pipeline 能分辨实例当前使用的 Java 来自用户配置（用户环境）还是 bundle（pipeline 托管）
2. 当实例使用 bundle JRE 时，runtime stage 无论定位是否成功，都重新校验 bundle JRE 的完整文件清单；损坏或缺失的文件按现有 hash 校验机制重下
3. 自愈对单个文件生效——只重下校验失败的文件，不整体重装

**非目标**

- 不改动用户自配 JRE 的处理方式——不校验、不修复、不探测，完全交给用户环境
- 不替换现有的 Java 定位与首次安装流程
- 不引入"运行 java 取版本号"之类的运行时探测作为自愈判据

**不做的事**

- 不在 Java 定位调用点里塞运行时探测逻辑——定位是轻调用，职责是"找到路径"而非"验证可用"
- 不把用户 JRE 的文件纳入 pipeline 的校验/重下范围

## 关键决策与取舍

### 决策 1：以"JRE 归属"为自愈触发条件，而非"探测可用性"

自愈该不该跑，看 Java 是谁的：bundle 的归 pipeline 管，必校验；用户的不归 pipeline 管，跳过。

**备选 A（否决）：在定位调用点加运行时探测（跑一次 java 判能否启动），起不来就当没找到、触发重装。** 否决理由：定位调用点是轻量职责，塞进程探测混淆了"定位"与"验证"；且探测只能给出"能不能跑"的二值结论，不能定位是哪个文件坏了，只能整体重装，与 pipeline 其余部分"逐文件 hash 校验、只重下坏的"的精度不一致。

**备选 B（否决）：不分归属，把 Java runtime 文件无条件纳入固化阶段的清单校验。** 否决理由：用户 JRE 的文件在用户系统里，pipeline 无权校验或改写，无条件校验会越界触碰用户的系统文件。

### 决策 2：bundle JRE 的自愈复用现有文件清单校验机制

bundle JRE 本身就有一份完整的文件清单（首次安装时从运行时清单获取的逐文件路径/地址/hash）。自愈 = 重新跑一遍这份清单的 hash 校验，不匹配的重下——和 library/asset 走的完全是同一套，不新造校验路径。

## 影响面

- **Java 定位层** —— 需要具备分辨并传达"命中的 Java 来自用户配置还是 bundle 目录"的能力，供下游 stage 决策
- **runtime stage** —— 从"找不到才下载"改为"bundle JRE 必校验清单、用户 JRE 直接跳过"
- **部署上下文** —— 可能需要承载当前 Java 的归属信号，在 stage 间传递
- **不影响** —— library/asset/package 的现有自愈流程、启动命令装配、首次安装流程、用户 JRE 的任何行为

## 验收标准

| 场景                     | 期望行为                             |
|------------------------|----------------------------------|
| bundle JRE 的任一文件被截断/损坏 | 启动实例时 pipeline 检测到、只重下该文件、实例正常启动 |
| 用户自配 JRE 损坏            | pipeline 不干预，行为与现状一致             |
| 首次安装 bundle JRE（无缓存）   | 正常下载、正常启动                        |
| bundle JRE 完好          | 启动时清单校验全过、不重下、启动耗时不显著增加          |
| 删除整个 bundle JRE 目录     | 等价于现有"找不到"行为，正常重装                |

## 备选方案备案

- **在定位调用点加运行时探测** —— 见决策 1：职责混淆 + 只能整体重装，精度不如逐文件校验
- **无条件把 runtime 纳入清单校验（不分归属）** —— 见决策 1：越界触碰用户系统文件
- **把 Java 定位委托异步化以便调用现有异步探测器** —— 同样落入"探测可用性"路线，且委托签名变更波及所有调用点，收益不抵成本
