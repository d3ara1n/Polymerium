## 计划

### 背景与动机

实例部署同时面对三种状态：由 packages、live 与 persist 计算出的当前目标状态，上一次成功部署形成的已应用状态，以及 build 目录此刻真实存在的状态。当前符号链接方案让第三方文件在 build 中直接指向来源，借此同时承担内容投影和来源记录两项职责。

这种职责耦合使 build 的实际状态被误当成上一次部署状态。只要游戏、用户或外部工具改写 build，规划阶段看到的内容就已经混入漂移，无法先得到纯粹的“上一次部署与本次目标之间的差异”，再判断该差异能否安全应用到实际目录。符号链接同样可能被删除、替换、改向或被目标程序拒绝；它没有消除冲突，只是让冲突提前污染部署规划。

目录型资源包暴露了这种设计的兼容性问题：游戏可以接受资源包和允许的符号链接路径，却不保证包内每个符号链接文件都参与资源枚举。继续按文件类型或目录位置增加例外只能缓解单个症状，无法解除部署状态追踪对符号链接语义的依赖。

部署模型需要明确分离三件事：

- 上一次成功部署承诺了什么；
- 当前根据来源计算后应该得到什么；
- build 中现在实际有什么。

部署差异只由前两者决定，实际状态仅在应用差异时用于校验前置条件、识别漂移和处理冲突。

### 目标

- 为每个实例保留独立于 build 的上一次成功部署快照，使来源消失后仍能判断哪些旧文件属于部署引擎并可被清理。
- 由上一次已应用快照和当前目标快照生成部署差异，不让 build 中的用户修改、游戏输出或异常残留进入差异规划。
- 将差异应用到 build 时显式比较实际状态，区分未变化、缺失、漂移、路径占用和非托管内容。
- 第三方文件以普通实体文件呈现给游戏，不再依赖符号链接表达来源。
- 包管理器管理的不可变包文件继续允许使用符号链接，以保留低成本投影、明确来源和快速重新布局的优势。
- 让两种投影方式共享同一套部署规划、冲突语义和成功提交边界，而不是继续形成两套互不一致的部署逻辑。
- 在异常中断、状态记录缺失或损坏、用户直接修改 build 等情况下保持保守且可恢复，不因无法证明所有权而误删数据。

### 非目标

- 不把 build 变成新的事实来源；packages、live 与 persist 仍然共同决定目标状态。
- 不阻止用户或游戏修改 build，也不承诺让 build 在运行期间始终与已应用快照一致。
- 不在本任务中重新定义 import、live 与 persist 各自承载的数据生命周期和优先级；只要求部署状态追踪尊重既有语义。
- 不为所有运行时输出建立所有权记录。日志、崩溃报告、截图、世界运行数据等仍应按其既有持久化或非托管规则处理。
- 不以解决某一种资源、某一个文件扩展名或某一个游戏版本为边界；资源包音乐只作为暴露架构问题的验收场景。
- 不依赖操作系统专有文件元数据、文件系统驱动或虚拟文件系统提供来源追踪。

### 核心状态模型

部署引擎以三个彼此独立的状态工作。

#### 上一次已应用快照

它表示上一次完整成功的部署向 build 承诺并实际交付过的托管路径集合。每个条目至少要表达目标路径、来源身份、部署时内容指纹、所有权类别和变更处理策略。

这份快照不是 build 当前状态的缓存，也不因 build 后来被修改而失效。它回答的是“部署引擎上次放置了什么”，因而是识别漂移、判断旧文件是否可删除以及处理中断恢复的基准。

#### 当前目标快照

它由当前 packages、live 与 persist 按既有优先级和冲突规则计算，表示本次部署完成后应由部署引擎管理的路径及内容。它不能从 build 反推，也不能为了迎合 build 中的异常内容而隐式改变。

#### 当前实际状态

它是应用部署差异时观察到的 build 文件系统状态。实际状态可以偏离上一次已应用快照，包括文件被删除、被修改、由文件变成目录、由托管项变成其他来源，或出现从未被部署引擎管理的占用项。

实际状态只参与应用校验和冲突处理，不参与生成上一次状态与当前目标之间的差异。

### 部署阶段边界

一次部署在语义上分为四个阶段。

#### 运行时变更收集

对允许游戏写回的托管内容，先以上一次已应用快照为基准识别 build 中的变化，并按既有数据归属规则回收到 live 或 persist。这个阶段是把允许的运行时变化纳入事实来源，不是让 actual 直接参与部署差异。

不可变内容的变化不自动回收到来源，而是记为漂移，留给冲突策略处理。来源已经消失或无法确认归属的变化必须保留到恢复区域或要求用户决策，不能为了完成部署而静默丢弃。

#### 目标计算与差异规划

运行时变化收集完成后，由事实来源计算当前目标快照，再与上一次已应用快照比较。规划结果只描述托管路径需要新增、替换、保留、移除或改变所有权，不包含 actual 中的偶发内容。

#### 差异应用

每项计划操作在修改 actual 前，都要验证该路径是否仍满足上一次已应用状态所给出的前置条件。满足时执行计划；不满足时按漂移或占用冲突处理，而不是重新规划为“actual 看起来想要的状态”。

#### 成功提交

只有全部必要操作完成并验证后，当前目标快照才成为新的已应用快照。失败或取消不得把尚未完整交付的目标声明为成功状态。下一次部署必须能够识别未完成的一代并恢复、回退或重新应用，而不是把中间状态误认为用户文件。

### 两种投影策略

投影策略由来源所有权和可变性决定，不由单个扩展名或某次兼容性问题决定。

#### 包文件投影

包管理器控制、内容不可变、能够从仓库或缓存重新取得的包文件可继续使用符号链接。符号链接在这里仅是交付方式，不再兼任部署规划的唯一历史记录。

包投影仍纳入统一的已应用快照，使部署差异保持独立于 actual。符号链接目标可作为应用阶段验证来源身份和诊断异常的额外证据，但不能替代上一次已应用快照。被改向、替换或占用的包路径按不可变托管内容处理。

#### 第三方文件投影

来自整合包覆盖内容、live 或 persist 的第三方文件以普通实体文件部署。其来源、上次交付内容和所有权完全由已应用快照表达，不再从文件类型、符号链接目标或操作系统附加属性推断。

第三方目录中的压缩包、音频、贴图、配置或其他文件遵守同一原则；是否实体化由来源类别决定，而不是由游戏当前是否能够读取某类符号链接决定。

### 所有权与变更策略

已应用快照中的托管条目需要按行为区分，而不是只记录“存在过”。

#### 不可变托管内容

包文件和其他声明为不可修改的部署资产属于此类。实际内容与上次部署不一致时视为漂移，不自动写回来源。仍被目标需要时恢复目标内容；不再需要时，漂移副本不能静默删除，应保留供恢复或产生清晰冲突。

#### 可写回托管内容

预期由游戏或用户在 build 中修改、并应成为后续目标状态一部分的内容属于此类。实际内容与上次部署不一致时，在计算目标前按既有归属规则回收到 live 或 persist。回收成功后，新的来源状态参与目标快照计算。

#### 非托管内容

没有出现在上一次已应用快照、也不属于当前目标的 actual 内容不归部署引擎所有。正常部署不得删除、移动或收编它。

如果非托管内容占用了当前目标路径，则形成路径占用冲突。只有内容与目标完全一致且采用明确、安全的收编规则时，才可将其纳入新的已应用快照；否则必须保留原内容并报告冲突，或经用户选择后移入恢复区域。

### 漂移与冲突处理

用户修改 build 不会使已应用快照失效，而会形成“actual 与 previous 不一致”的漂移。处理结果由 previous、desired 和 actual 三方关系决定。

| 上一次已应用状态 | 当前目标状态 | 当前实际状态 | 处理结果 |
| --- | --- | --- | --- |
| 有 | 无 | 与上次一致 | 安全删除托管项 |
| 有 | 无 | 已修改 | 不直接删除，按策略写回、保留或报告冲突 |
| 有 | 有且内容不变 | 与上次一致 | 保留，无需重写 |
| 有 | 有且内容变化 | 与上次一致 | 应用目标更新 |
| 有 | 有 | 实际缺失 | 重新创建目标内容 |
| 有 | 有 | 已修改 | 先按可变性处理漂移，再恢复目标或报告冲突 |
| 无 | 有 | 不存在 | 创建并纳入新快照 |
| 无 | 有 | 已有等同内容 | 可按安全收编规则纳入新快照 |
| 无 | 有 | 被不同内容或不同节点类型占用 | 不静默覆盖，报告或隔离冲突 |
| 无 | 无 | 存在 | 视为非托管内容并保持不动 |

路径节点类型变化同样属于冲突。文件、目录和符号链接之间的替换不能仅凭路径相同而视为内容更新。

### 删除安全性

部署引擎只能删除能够由上一次已应用快照证明所有权、且 actual 仍与上次交付状态一致的内容。

来源当前是否仍存在不影响这项证明。即使某个来源已从 packages、live 或 persist 消失，只要已应用快照证明该 actual 条目仍是上次未修改的部署产物，就可以安全删除；如果 actual 已漂移，则必须先保护变化。

目录清理只能移除由托管项清空后确认不再包含非托管内容的空目录。目录本身不能因为曾由部署引擎创建，就连带删除后来由游戏或用户放入的未知文件。

### 已应用快照的可靠性

已应用快照必须满足以下语义约束：

- 独立于 build 保存，删除或重建 build 不会连带删除唯一的部署历史。
- 以成功部署代次为边界，能够识别它对应哪一次完整交付。
- 记录足以验证内容是否仍与上次交付一致的稳定指纹，不以修改时间作为唯一依据。
- 保存逻辑路径和来源身份，不依赖某个平台上的临时绝对路径保持永久不变。
- 更新具备原子成功语义，不出现一半旧状态、一半新状态被声明为有效的情况。
- 状态缺失、损坏或与实例代次不匹配时进入保守模式，不执行无法证明所有权的删除。

### 状态缺失与恢复

当已应用快照不可用时，普通实体文件无法自行证明来源。部署引擎必须承认这一信息边界，不通过路径名称、内容相似或目录位置武断恢复所有权。

恢复遵循以下原则：

- 与当前目标内容完全一致的 actual 条目可以按明确规则收编，以减少不必要的重写。
- 无法证明所有权的多余 actual 条目保持非托管，不自动删除。
- 需要占用已有未知内容的目标路径形成冲突，不自动覆盖。
- 用户可以选择清理并重新生成整个可丢弃的 build，从当前事实来源建立新的已应用快照。
- 旧符号链接方案迁移时，可利用仍然有效且指向受控来源的链接建立初始来源关系；损坏、改向或来源不可信的链接不能被自动认定为安全托管项。

### 从现有部署模型迁移

迁移需要在不丢失用户状态的前提下，把第三方文件从链接投影转成实体投影，并建立首份已应用快照。

- 包管理器控制的不可变包链接保持原有交付方式，但纳入统一部署状态语义。
- 能够验证来源身份的第三方链接可作为迁移输入，实体化后由首份已应用快照接管来源追踪。
- 已被普通文件替换的旧链接位置按潜在漂移处理，不能当作干净部署产物覆盖。
- build 中无法关联到受控来源的普通文件保持非托管。
- 迁移完成后，第三方实体文件的清理完全依据已应用快照，不继续依赖残留链接或兼容性例外。
- 首次迁移失败时不得留下“已迁移”的成功状态；后续运行应能够安全重试。

### 硬链接定位

硬链接不作为来源追踪机制。它只能证明两个已知路径在当前时刻共享同一文件实体，不能从一个路径可靠反查来源；来源路径删除后关系信息会弱化，目标被删除后重建会断开关系，原地修改还可能污染来源。

硬链接也不适合作为第三方实体投影的默认交付方式，因为 build 中允许发生的修改不应隐式修改 live、persist 或缓存。未来若将硬链接用于明确不可变、可校验的局部性能优化，也不能替代已应用快照或改变冲突语义。

### 影响面

本任务会影响以下能力领域：

- 实例目标状态的生成与来源优先级整合；
- 部署差异规划和实际文件系统应用之间的边界；
- 第三方文件、包文件和持久化文件的投影策略；
- 游戏运行后变更的检测与写回；
- 更新、重置、重新部署和失败恢复；
- 路径冲突、用户修改和未知文件的诊断呈现；
- 实例删除、快照恢复及 build 重建时的部署状态生命周期；
- 旧实例从文件级符号链接投影迁移到实体投影的兼容流程。

### 验收标准

#### 正常部署

- 新实例完成部署后，第三方文件在 build 中均为游戏可直接读取的普通实体文件。
- 包管理器控制的不可变包仍可使用符号链接，且重新布局后旧包链接会被准确移除、新包链接会被补全。
- 目录型资源包及其音频、贴图、模型和语言资源能按游戏原生文件语义加载，不依赖包内文件符号链接兼容性。
- 上一次成功部署状态与 build 分离保存，清空可丢弃的 build 后可以从事实来源完整重建。

#### 更新与删除

- 第三方来源移除后，未被修改的旧部署文件会从 build 中删除。
- 第三方来源新增或改变后，build 会得到对应实体内容，已应用快照在成功后反映最终状态。
- 来源消失但 build 文件已被修改时，部署不会静默删除修改内容。
- 托管目录内后来出现的非托管文件不会因清理旧来源而被连带删除。

#### 漂移与冲突

- 用户删除托管文件后，下一次部署可依据目标状态重建它。
- 用户修改可写回内容后，变化会按既有归属进入 live 或 persist，并参与后续目标计算。
- 用户修改不可变内容后，部署会明确识别漂移，并按保护策略处理，而不是把它悄悄当作新的目标状态。
- 未受管理的文件占用目标路径时，部署不会静默覆盖或删除它。
- 符号链接被改向、替换或损坏时，包投影同样进入冲突处理，不因使用链接而绕过 actual 校验。

#### 可靠性与恢复

- 部署中断不会把未完整应用的目标状态提交为新的成功快照。
- 下一次运行能够识别并处理上次中断留下的部分 actual 变化。
- 已应用快照缺失或损坏时，部署默认不删除无法证明所有权的实体文件。
- 用户执行明确的 build 重建后，可以从当前 packages、live 与 persist 建立干净且一致的新快照。
- 旧实例迁移不丢失已经偏离链接来源的用户文件，也不会把未知普通文件误认成可删除的托管内容。

### 备选方案备案

#### 全面保留符号链接并继续增加例外

不采纳。符号链接同时承担内容投影和状态追踪，导致 previous 与 actual 无法分离；游戏和平台对链接的支持差异还会持续产生新的例外。允许列表、目录级链接或特定文件实体化都只能修复局部兼容性，不能解决部署规划被漂移污染的问题。

#### 第三方文件使用硬链接代替符号链接

不采纳。硬链接不能可靠表达来源路径，来源删除或目标重建后无法继续证明所有权，且 build 的原地修改可能反向污染事实来源。

#### 通过扩展属性或平台专有附加数据标记来源

不采纳。附加元数据在不同平台、复制工具、压缩流程和安全软件之间缺乏稳定保证，丢失后仍无法安全删除，不能成为部署所有权的唯一依据。

#### 从当前来源和 actual 内容反向推断上次部署状态

不采纳。当前来源只能表达 desired，无法证明来源已经消失的 actual 条目是否曾由部署引擎创建；内容相同也不能证明所有权。反向匹配只能用于保守收编，不能替代 previous。

#### 将所有 actual 内容都视为托管并按目标快照强制同步

不采纳。build 中包含游戏与用户产生的合法内容，强制镜像会删除未知数据并掩盖路径冲突。

#### 为每个第三方文件放置同目录旁车标记

不采纳。旁车文件会污染游戏目录、可能被游戏扫描或被用户删除，并把一个文件的状态拆成多个容易失配的目录项。实例级已应用快照更适合表达一代部署的整体承诺。

#### 完全取消包文件符号链接

暂不采纳。包文件内容不可变、来源受包管理器控制且可重新取得，链接在这一边界内仍能提供明确且低成本的投影。计划要求把链接降级为交付策略并纳入统一的 previous、desired、actual 语义；若未来仍出现不可接受的跨平台兼容性，可在不改变部署状态模型的前提下替换包文件的交付方式。
