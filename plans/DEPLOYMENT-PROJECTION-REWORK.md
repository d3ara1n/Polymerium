> 制定日期：2026-07-15
> 定位：重构部署管线的投影收集与应用机制——按交付语义切分通道，并为 import 实体投影引入与软链接对账器同形的对账式交付
> 当前状态：未实施

## 计划

### 背景与动机

`IMPORT-ENTITY-PROJECTION.md` 把 import 的投影从软链接改为实体入 build，并要求 import 的实体投影能对账——快照 build 中当前的 import 投影状态、按期望集补缺与（更新时）清除。现有的固化步骤是一个把多种文件混在一起处理的单体逻辑：下载到缓存的库与资源、解压入 build 的 natives、软链接入 build 的包与持久化、复制进工作副本的 import，全都挤在同一步里就地处理，且投影优先级仲裁也内联其中。

import 改实体入 build 后，它需要的对账语义（跨多次部署比对、补缺、留旧、清废）与下载、解压那种「一次性产出」的语义完全不同；继续塞在单体固化逻辑里，既无法独立对账，也会让固化步骤职责膨胀。管线需要按交付语义切分通道，让 import 的实体投影拥有一个与软链接对账器同形的、独立的对账式交付通道。

### 目标

- 部署管线的投影应用按交付语义切分为清晰通道：软链接（包、持久化）、下载到缓存（库、资源、运行时）、解压入 build（natives）、实体入 build（import）。
- import 实体投影以对账式交付：能快照 build 中当前的 import 投影状态，按期望集对账——日常部署补缺失、留已存在；与软链接对账器形态一致。
- 跨通道的优先级仲裁（持久化 > import > 包）一致执行：同一路径被持久化与 import 同时声明时，持久化软链接占据，import 实体不放置。
- 遗留的软链接式 import 投影在 import 通道对账时被识别并报错。

### 非目标 / 不做的事

- 不改下载到缓存通道与解压通道的语义——库、资源、运行时仍在全局缓存、不进 build；natives 仍解压进 natives 目录。
- 不重新定义 import 投影模型本身（投影对象、优先级顺序、替换式更新、工作副本语义）——那是 `IMPORT-ENTITY-PROJECTION.md` 的范围；本计划只负责管线如何交付与对账。
- 不在本计划落实工作副本、快照、workspace 侧的改造——它们依赖本计划提供的对账能力，但归属 `IMPORT-ENTITY-PROJECTION.md`。
- 不为旧软链接式 import 投影做自动迁移——检测到即报错，由重置清理。

### 关键决策与取舍

- **按交付语义切分通道，而非按文件来源。** 软链接、下载缓存、解压、实体入 build 各有不同的产出与对账语义；按交付方式切分能让每个通道语义纯粹。按来源切分（包/库/资源/import）会把语义不同的产出混进同一通道。
- **import 实体投影走独立对账式通道，与软链接对账器同形。** 实体文件不像软链接那样自描述来源，必须靠对账（快照现状 + 期望集比对）来决定补缺、留旧、清废；这与软链接对账器面对的问题同构，故取同形。
- **优先级仲裁跨通道一致执行，仲裁顺序不变。** import 从软链接通道迁出后，持久化与 import 分属不同通道，但同路径冲突的仲裁仍须统一裁决并保持原顺序。
- **import 实体投影不复用下载缓存通道。** import 内容本就在 import 目录、无需下载，且要的是 build 内实体文件，与「下载到全局缓存」语义不符。

### 影响面

- **投影固化机制** —— 由单体固化拆为按交付语义切分的多个通道；import 实体投影获得独立的对账式交付。
- **优先级仲裁** —— 从单体内联改为跨软链接通道与 import 实体通道的统一裁决。
- **遗留检测** —— import 通道对账时承担旧软链接式投影的识别与报错。
- **下载缓存通道与解压通道** —— 语义不变，仅从单体固化中剥离为独立通道。

### 验收标准

- 部署后，build 中各通道内容各归各位：包与持久化为软链接、库/资源/运行时在缓存（不在 build）、natives 解压在 natives 目录、import 为实体文件。
- import 实体投影对账：日常部署只补缺失、不动已存在；期望集中移除的项在更新后对账时被清除。
- 同一路径被持久化与 import 同时声明时，持久化软链接占据，import 实体不放置。
- import 投影对账与软链接对账各自独立，互不误删对方的内容。
- 检测到遗留的软链接式 import 投影时，对账报错并提示重置实例。

### 备选方案备案

- **把 import 实体投影留在现有单体固化逻辑里处理。** 否决。实体投影需对账语义（跨次部署比对、补缺、清废），与下载/解压的「一次性产出」语义不同；混在一起会让固化步骤既管下载又管对账，职责膨胀，且 import 无法独立对账。
- **import 实体投影每次全量重写、不对账。** 否决。会覆盖游戏/用户对工作副本的改动，违背「日常部署已存在不动」的模型约束。
- **import 实体投影复用下载缓存通道（下载到缓存再链入 build）。** 否决。import 内容本就在本地、无需下载，目标也是 build 内实体而非缓存引用，语义不符。

### 关联计划

投影模型与各受影响层（工作副本、快照、workspace、整合包更新、导出）的边界由 `IMPORT-ENTITY-PROJECTION.md` 固定；本计划只负责部署管线如何按通道交付与对账，不重新定义模型。
