## 支持的格式

Polymerium 支持四种用于导入和导出的整合包格式：

| 格式 | 扩展名 | 说明 |
|--------|-----------|-------|
| **Trident**（原生） | `.zip` | 保留最多元数据：标签、规则、覆盖、图标、README、更新日志 |
| **CurseForge** | `.zip` | 行业标准；兼容 CurseForge 应用、Prism 等 |
| **Modrinth** | `.mrpack` | 现代格式；兼容 Modrinth 应用、Prism 等 |
| **MultiMC** | `.zip` | 兼容 MultiMC、Prism Launcher 及其衍生版本 |

## 导入整合包

1. 点击 **+** 按钮创建新实例。
2. 选择**从文件导入**。
3. 选择整合包存档。
4. Polymerium 自动检测格式并导入：

   - 元数据被解析并写入实例的 `profile.json`。
   - 整合包文件（配置、资源）被提取到 `import/`。
   - 整合包中列出的模组会被解析并下载到共享缓存中。

5. 部署并游玩。

> **更新整合包？**如果实例是通过导入创建的，您可以重新导入同一整合包的较新版本。Polymerium 会用新文件替换 `import/`，同时保持 `persist/` 不变 — 您移动到那里的文件将被保留。

## 导出实例

1. 选择一个实例。
2. 打开实例菜单 → **导出**。
3. 选择格式，填写元数据（名称、作者、版本、摘要）。
4. Polymerium 将实例元数据 + `import/` 文件打包为所选格式。

### 导出内容

| 内容 | 是否包含 |
|---------|:---------:|
| 模组列表 + 版本 | ✅ |
| 配置文件（`import/`） | ✅ |
| 实例图标 | ✅ |
| 整合包元数据（名称、作者、版本） | ✅ |
| 存档、截图、个人配置 | ❌ — 如果您将这些移动到 `persist/`，它们会保留在 `persist/` 中，不会包含在导出中 |

### 格式特定说明

- **Trident 格式**保留最多信息：标签、部署规则、账号绑定和额外文件。最适合 Polymerium 之间的共享。
- **CurseForge / Modrinth** 格式广泛兼容但会丢失 Polymerium 特定功能，如标签和规则。
- **MultiMC 格式**是最基础的 — 仅模组列表和配置。

## 在线与离线格式

并非所有整合包格式在底层的工作方式都相同：

- **CurseForge 和 Modrinth** 格式是**在线**格式 — 它们包含托管在这些平台上的模组的引用（包 URL），而不是实际的模组文件。导入时，启动器会解析这些引用并下载模组。

  这意味着 CurseForge 整合包理想情况下应仅引用 CurseForge 模组，Modrinth 整合包应仅引用 Modrinth 模组。如果您混合来源，整合包在发布时将无法通过平台审核。

- **MultiMC** 格式是**离线**格式 — 它将实际模组文件打包在存档中。导入时无需互联网解析。

- **Trident** 既非纯在线也非纯离线。默认使用在线引用（PURL）并在部署时解析模组，但可选择离线模式将模组文件打包。详见下方 [Trident 便携实例](#trident-便携实例) 章节。

在为发布您的整合包选择格式时请记住这一点。

## Trident 便携实例

Trident 格式与其他整合包格式有本质区别。它不仅仅是一个模组列表，而是一个**完整的实例快照**——Polymerium 对某个实例所知的一切的可移植表示。

### 包含内容

一个 Trident `.zip` 存档包含：

| 文件 | 用途 |
|------|---------|
| `trident.index.json` | 完整实例配置：含 PURL 的模组列表、标签、规则、覆盖、来源追踪 |
| `trident.options.json` | 导出配置（离线模式、排除的标签、包含的覆盖项） |
| `icon.png`（或其他格式） | 实例图标——存在时自动打包 |
| `README.md` | 实例说明文档 |
| `CHANGELOG.md` | 实例更新日志 |
| `LICENSE.txt` | 实例许可证 |
| `import/` | 覆盖文件（配置、脚本、自定义资源） |

### 丰富的元数据

Trident 保留了其他格式无法承载的数据维度：

- **每个模组的启用/禁用状态**——不仅知道有哪些模组，还知道哪些处于激活状态
- **标签**——用户自定义的分类标签
- **部署规则**——按模组的覆盖设置（目标路径、跳过、标准化）
- **实例覆盖**——JVM 参数、窗口大小、服务器地址、部署策略
- **来源追踪**——每个模组的来源信息，在导出/导入中完整保留

这使得 Trident 成为唯一能忠实复现实例完整配置的格式——而不只是模组列表。

### 默认在线，按需离线

Trident 是一种**混合**格式：

- **在线模式（默认，推荐）**：模组以 PURL 引用形式存储（如 `modrinth:fabric-api/P7uGFii0`）。实际的 `.jar` 文件在部署时解析并下载。存档体积小，适合版本控制。
- **离线模式（可选）**：所有已启用模组的 `.jar` 文件打包进存档。适用于归档或无网络环境。

日常使用推荐在线模式：存档更小，导入时模组更新鲜，且 `trident.index.json` 可读性好、方便比较差异。

### 实例附属文件

与其他整合包格式不同，Trident 将实例级别的文档文件视为格式的一等公民。`icon.png`、`README.md`、`CHANGELOG.md` 和 `LICENSE.txt` 只要存在于实例主目录中，就会自动打包。这些文件描述的是**实例本身**——而非游戏——并在导出/导入过程中完整保留。

### 理想的实例模板

由于 Trident 捕获了完整的实例配置，它非常适合作为**实例模板**。导出一个配置完善的实例，然后反复导入即可批量创建配置统一的预制实例。标签、规则、覆盖和文档全部忠实转移——无需手动重新配置。
