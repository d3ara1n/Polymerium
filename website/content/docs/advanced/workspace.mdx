## Who Is This For?

The Workspace is built for **modpack developers** — people who create and maintain modpacks.

If you imported someone else's modpack, you don't need this feature. Your `import/` folder is effectively the pack source, and the game runs from `build/`. The only time you might open the workspace is to restore an original file from `import/` into the working copy if something broke — but that's rare.

## The Problem

When you develop a modpack, you configure things in-game: video settings, mod configs, world generation options. Those changes land on the **working copy** — the real files under `build/` that were projected from `import/`. The `import/` tree itself stays untouched. You changed configs but don't know which files moved or how to promote them back into the modpack. That's what Workspace solves.

## Opening the Workspace

1. Select your instance.
2. Switch to the **Workspace** tab.

The workspace compares `import/` (source) with the import projection in `build/` (working copy). Only real files owned by that projection are listed — package and `persist/` symlinks are skipped.

## Understanding the Diff

The workspace scans the `import/` manifest and checks the matching path under `build/`:

| Type | Meaning |
|------|---------|
| **Updated** | Working copy is newer than `import/` (typical after in-game edits) |
| **Outdated** | `import/` is newer than the working copy |
| **Same** | No difference (not listed) |

Files that exist only under `build/` and were never declared in `import/` are outside the working copy and do not appear here.

### Example: Config Tweaks

You launched the game and adjusted performance settings for your modpack. The changes are in `build/config/` while the originals sit in `import/config/`. The workspace shows exactly which import-owned files changed and how they differ.

## Syncing Changes

There are two sync directions, each serving a different purpose.

### Promote: working copy → import (primary)

This is the main workflow. When you see a change in the working copy that you want to keep in your modpack:

1. Select the file in the workspace.
2. Click **Sync to Import**.
3. The file is copied from `build/` into `import/`.

That change is now part of your modpack's source. It will be included when you export or update the modpack.

You can select multiple files and sync them at once — useful after a tuning session where you changed many config files.

### Restore: import → working copy (secondary)

If an in-game change went wrong and you want the original file back:

1. Select the file in the workspace.
2. Click **Sync to Live** / restore.
3. The file is copied from `import/` back onto the working copy in `build/`, overwriting your local change.

This restores the modpack's original version of that file.

## Modpack Author Workflow

This is the primary use case for the workspace:

1. Build a modpack and add initial configs to `import/`.
2. Launch the game and configure everything in-game (changes go to the working copy in `build/`).
3. Open the workspace and review what changed.
4. Promote confirmed changes from the working copy to `import/`.
5. Export or commit to Git.

Repeat this loop as you iterate on your modpack.

## Relationship to Git

If you use Git for your modpack (see [Modpack Development](/docs/advanced/modpack-dev)), the workspace fits naturally into version control:

1. `import/` is tracked by Git.
2. `build/` and `persist/` are in `.gitignore`.
3. Promote from working copy → `import/` → then `git commit`.

Think of the working copy in `build/` as your working tree, `import/` as your staged changes, and `git commit` as the final step.
