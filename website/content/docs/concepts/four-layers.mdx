## The Layers

Every instance in Polymerium has three directories on disk. Each has a different job and a different lifecycle. Runtime changes to imported content do not live in a separate folder — they land in `build/` as the **working copy** of what `import/` declared.

```
instances/my-modpack/
├── import/     ← Files that ship with the modpack (authoritative source)
├── persist/    ← Files you choose to preserve across resets and updates
└── build/      ← The deployed game directory; also holds runtime mutations to imported content
```

### `import/` — The Modpack Source

This is what the modpack author intended. When you import a modpack, its files land here — configs, default options, custom resources, etc.

On deploy, Polymerium projects `import/` into `build/` as **real files** (not symlinks). That lets the game read directory-style resource packs and similar content that would not follow a symlink.

- **Who writes it:** The modpack author (via import or manual placement)
- **When it changes:** When you update or re-import the modpack, or when you promote changes from the workspace
- **What it's for:** The authoritative, distributable version of the modpack's files

### Working copy — Import projected into `build/`

There is no separate `live/` directory. The working copy is a **virtual layer**: its file list comes from `import/`, and its on-disk state is the matching real files under `build/`.

When you play, in-game edits to imported content (modified configs, generated files under paths that came from `import/`) update those files in `build/`. The Workspace compares that projection against `import/` so you can promote or restore changes.

- **Who writes it:** The game runtime (and you, when tweaking configs)
- **When it changes:** Every play session that touches imported paths
- **What it's for:** A sandbox for runtime modifications without polluting the source in `import/`

Paths that `persist/` or packages win on are symlinks in `build/` and are **not** part of this working copy.

### `persist/` — User-Managed Persistence

This directory is **yours**. Polymerium doesn't write to it automatically — you decide what goes here.

The idea is simple: some files are too important to lose on reset or modpack update. Maybe your `options.txt` has sensitivity settings you spent an hour dialing in, or your `saves/` directory has a world you can't afford to lose. You move those files into `persist/` yourself.

**How it works:**

1. **For single files** — move the file from `build/` to `persist/`. For example, move `build/options.txt` to `persist/options.txt`. On the next deploy, Polymerium will link `persist/options.txt` into `build/` with the highest priority, so your version always wins.

2. **For directories** — move the directory (e.g. `saves/`) into `persist/`, then create an empty `.keep` file inside it: `persist/saves/.keep`. This tells Polymerium to project the entire directory into `build/`.

- **Who writes it:** You
- **When it changes:** When you decide to add or update files
- **What it's for:** Files you want to survive resets, modpack updates, and rebuilds

### `build/` — The Deployed Game

This is the final game directory — what Minecraft sees as `.minecraft`. It's assembled from packages, `import/`, and `persist/` during deploy.

- **Who writes it:** Polymerium's deploy engine, plus the game at runtime
- **When it changes:** Every deploy, and while you play
- **What it's for:** The runnable game directory
- **Can you delete it?** Yes — but anything that only exists in `build/` and isn't backed by `persist/` or `import/` will be lost. Files you've moved to `persist/` are always safe. Package content and the rest of the tree get rebuilt from metadata and cache.

## How They Relate

```
import/  ──(real files)──→  build/   ← game also writes imported paths here
persist/ ──(symlinks)────→  build/
packages ──(symlinks)────→  build/
```

During deploy, content is layered into `build/` in this order:

1. **Cached packages** — mods, libraries (symlinked from cache)
2. **`import/` files** — modpack source, copied as real files (fill-missing only on normal deploy)
3. **`persist/` files** — your preserved files (symlinked, highest priority, always wins)

If the same path is claimed by both `import/` and `persist/`, the `persist/` version takes precedence in `build/`. Package symlinks lose to both.

## What Happens During Common Operations

| Operation | `import/` | Working copy in `build/` | `persist/` | `build/` (rest) |
|-----------|:---------:|:------------------------:|:----------:|:---------------:|
| **Deploy** | Read | Fill missing import files | Read (symlink) | Rebuilt as needed |
| **Play game** | Unchanged | Modified | Unchanged | Modified |
| **Reset** | Unchanged | Cleared with `build/` | Unchanged | Cleared |
| **Modpack update** | Replaced | Cleared then reprojected | Unchanged | Redeployed |

### Key Takeaways

- **Reset** clears `build/` (including the import working copy). Files you've placed in `persist/` are untouched.
- **Modpack update** replaces `import/` and clears the previous import projection in `build/`. `persist/` is preserved — that's the whole point.
- **`build/`** is disposable as long as anything you care about is backed by `persist/` (or still present in `import/`).
- Daily redeploy does **not** delete orphan files left after you hand-edit `import/`; if deploy fails because a real file blocks a package symlink, reset the instance.

## Why This Matters

This model solves a real tension in modded Minecraft:

- **Modpack authors** want to distribute consistent configs and resources.
- **Players** want to keep their settings and saves across updates.
- **Both** want modpack updates to work without losing important data.

Polymerium's answer: separate these concerns into distinct directories (and one virtual working copy) with different lifecycles. You control what's preserved — move it to `persist/`, and Polymerium guarantees it survives.
