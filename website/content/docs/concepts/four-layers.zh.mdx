## 分层

Polymerium 中的每个实例在磁盘上有三个目录。每个目录任务和生命周期不同。对导入内容的运行时修改不会单独占一个文件夹——它们落在 `build/` 里，作为 `import/` 声明内容的**工作副本**。

```
instances/my-modpack/
├── import/     ← 随整合包附带的文件（权威来源）
├── persist/    ← 你选择在重置和更新之间保留的文件
└── build/      ← 部署出的游戏目录；也承载对导入内容的运行时变更
```

### `import/` — 整合包来源

这是整合包作者的意图。当你导入整合包时，它的文件会放在这里——配置、默认选项、自定义资源等。

部署时，Polymerium 把 `import/` 以**实体文件**投影进 `build/`（不是软链接）。这样游戏才能直接读取目录型资源包等不会跟随软链接的内容。

- **谁写入它：** 整合包作者（通过导入或手动放置）
- **它何时改变：** 更新或重新导入整合包时，或你通过工作区把变更提升回去时
- **它的用途：** 整合包文件的权威、可分发版本

### 工作副本 — `import/` 在 `build/` 上的投影

没有独立的 `live/` 目录。工作副本是一层**虚拟层**：文件清单来自 `import/`，磁盘状态取自 `build/` 中对应的实体文件。

游玩时，对导入内容的游戏内修改（配置变更、落在 import 路径下的生成文件）会改写 `build/` 里的这些文件。工作区会把该投影与 `import/` 对比，方便你提升或还原。

- **谁写入它：** 游戏运行时（以及你调整配置时）
- **它何时改变：** 每次触及导入路径的游戏会话
- **它的用途：** 运行时修改的沙盒，不污染 `import/` 中的源

若某路径由 `persist/` 或包赢得投影，则 `build/` 上是软链接，**不属于**这份工作副本。

### `persist/` — 用户管理的持久层

这个目录是**你的**。Polymerium 不会自动写入它——你决定放什么。

想法很简单：有些文件太重要，不能在重置或整合包更新时丢掉。也许 `options.txt` 里有你调了很久的灵敏度，或者 `saves/` 里有不能丢的世界。你自己把它们移进 `persist/`。

**工作原理：**

1. **单个文件** —— 从 `build/` 移到 `persist/`。例如把 `build/options.txt` 移到 `persist/options.txt`。下次部署时 Polymerium 会以最高优先级把 `persist/options.txt` 链接进 `build/`，你的版本总是胜出。

2. **目录** —— 把目录（如 `saves/`）移进 `persist/`，再在里面放一个空的 `.keep`：`persist/saves/.keep`。这告诉 Polymerium 把整个目录投影进 `build/`。

- **谁写入它：** 你
- **它何时改变：** 当你决定添加或更新文件时
- **它的用途：** 希望在重置、整合包更新和重建中保留的文件

### `build/` — 部署的游戏目录

这是最终的游戏目录——Minecraft 看到的 `.minecraft`。部署时由包、`import/` 和 `persist/` 组装而成。

- **谁写入它：** Polymerium 的部署引擎，以及运行时的游戏
- **它何时改变：** 每次部署，以及游玩过程中
- **它的用途：** 可运行的游戏目录
- **你可以删除它吗？** 可以——但只存在于 `build/`、且没有 `persist/` 或 `import/` 支撑的内容会丢。你已放进 `persist/` 的文件始终安全。包内容与其余树会从元数据和缓存重建。

## 它们如何关联

```
import/  ──(实体文件)──→  build/   ← 游戏也会写入这些导入路径
persist/ ──(软链接)────→  build/
packages ──(软链接)────→  build/
```

部署时按此优先级写入 `build/`：

1. **缓存中的包** —— 模组、库（从缓存软链接）
2. **`import/` 文件** —— 整合包源，以实体复制（日常部署只补缺失）
3. **`persist/` 文件** —— 你保留的文件（软链接，最高优先级，总是胜出）

同一路径同时被 `import/` 与 `persist/` 声明时，`persist/` 优先。包软链接优先级最低。

## 常见操作期间发生什么

| 操作 | `import/` | `build/` 中的工作副本 | `persist/` | `build/` 其余部分 |
|------|:---------:|:--------------------:|:----------:|:-----------------:|
| **部署** | 读取 | 补齐缺失的 import 文件 | 读取（软链接） | 按需重建 |
| **玩游戏** | 不变 | 可能被修改 | 不变 | 可能被修改 |
| **重置** | 不变 | 随 `build/` 清除 | 不变 | 清除 |
| **整合包更新** | 替换 | 清除后重新投影 | 不变 | 重新部署 |

### 关键要点

- **重置**会清除 `build/`（含 import 工作副本）。`persist/` 中的文件不受影响。
- **整合包更新**会替换 `import/`，并清除 `build/` 中旧的 import 投影。`persist/` 会保留——这就是它存在的意义。
- 只要你在意的内容有 `persist/`（或仍在 `import/` 中），**`build/`** 就可以随时丢掉重建。
- 日常重新部署**不会**删除你手改 `import/` 后留下的孤儿实体；若实体文件挡住了包软链接导致部署失败，重置实例即可。

## 为什么这很重要

这套模型解决了模组 Minecraft 里的真实张力：

- **整合包作者**想分发一致的配置和资源。
- **玩家**想在更新之间保留设置和存档。
- **双方**都希望更新时不丢重要数据。

Polymerium 的做法：把这些关注点拆进不同生命周期的目录（以及一层虚拟工作副本）。你控制什么被保留——放进 `persist/`，Polymerium 保证它能留下来。
